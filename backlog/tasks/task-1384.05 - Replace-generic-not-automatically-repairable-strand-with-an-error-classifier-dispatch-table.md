## Problem
`repair-handoff.js:105-127` hard-codes only two repairable classes (dirty tree, behind remote) via string matching, then bails on everything else with "Handoff error is not automatically repairable", stranding the run. As new repair paths are added (gates, integrate-root, gatekeeper artifacts) this if/else does not scale.

## Fix
Introduce an error-classifier + dispatch table: each entry has a matcher (regex/predicate on the error message) and a repair handler. Iterate handlers; first match runs its repair (with bounded retries). Unmatched errors get a structured, actionable escalation message instead of a bare strand, including a one-line manual remediation (e.g. the `px review <task> --submit` hint already emitted).

## Acceptance
- Repair classes are registered declaratively, not hard-coded in one if-chain.
- Adding a new repairable class is a table entry + handler, no edits to the dispatch loop.
- Unmatched errors produce an actionable escalation message with manual next step.
- Tests for dispatch order, a matched repair, and an unmatched escalation.
