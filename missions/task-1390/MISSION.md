# Mission: Fix px shell-init bash broken after TypeScript conversion (task-1390)

## Goal

Restore `px shell-init bash` so that the emitted shell function produces a runnable `px` command when the package is published and installed globally. The root cause is that the TypeScript compilation pipeline (`build:cjs`) strips the shebang from `px.ts`, leaving `px.js` without a `#!/usr/bin/env node` header. This breaks direct execution of `px.js` and can cause the npm bin wrapper to behave unexpectedly in certain environments.

## Why Now

The TypeScript conversion replaced the previous JavaScript pipeline. The `build:cjs` command (`tsc --rootDir . --outDir . --module CommonJS --moduleResolution Node --esModuleInterop`) compiles `px.ts` to `px.js` at the package root, but TypeScript strips shebang lines during compilation (a known limitation documented in the TypeScript issue tracker). The `px.js` file is listed in `package.json` `"files"` and `"bin"` (`"px": "px.js"`), yet it arrives at the user's system without an executable shebang. The shell function generated by `px shell-init bash` uses `command px "$@"` which relies on the globally installed `px` binary working correctly — if the npm wrapper or the underlying `px.js` is broken, the entire shell integration fails.

## Scope

- Add a post-compilation step to `build:cjs` that restores the `#!/usr/bin/env node` shebang to the top of `px.js`
- Ensure `index.js` (compiled from `index.ts`) also retains its shebang if it has one
- Add a reproduction test that verifies `px.js` is directly executable with the shebang and that `node px.js shell-init bash` produces valid shell output
- Verify `npm pack` includes `px.js` and that a simulated global install works

## Out of Scope

- Changes to the `shellInit()` function template in `px.ts` (the shell function logic is correct; the problem is the runtime binary)
- Changes to the default `tsc` (ESM) build pipeline
- Changes to `package.json` bin configuration or files array
- Changes to `tsconfig.json` compiler options
- Fixes to `index.js` shebang unless proven necessary (only `px.js` is referenced by the npm bin entry)

## Success Criteria

> **Falsifiability rule (ADR 0039 Part 2):** Each criterion must be falsifiable.

1. `px.js` begins with the literal string `#!/usr/bin/env node` after `npm run build:cjs` completes. Verified by: `head -c 22 px.js` outputs exactly `#!/usr/bin/env node`.
2. Running `node px.js shell-init bash` exits with code 0 and the output contains the string `px() {` (the shell function header). Verified by: exit code check and `grep -c 'px() {'` returning 1.
3. The reproduction test in `test/task-1390-shell-init-shebang.test.js` passes green after the fix and fails red at the mission's parent commit.
4. `npm pack` produces a tarball that includes `px.js` with the shebang intact (verified by extracting the tarball and checking the first line).
5. All existing tests in `test/px-shell-init.test.js` continue to pass (no regression in shell-init behavior).

## Risks and Assumptions

- **Risk:** The post-compilation shebang restoration must run after `tsc` outputs `px.js`. If placed before or as part of the tsc command, it will not work. Assumption: a simple `sed` or `echo` prepend after `tsc` is sufficient.
- **Risk:** Modifying the build script may interfere with CI pipelines that expect the build output to be produced by `tsc` alone. Assumption: the CI runs `npm run build:cjs` which we control.
- **Assumption:** The `index.ts` shebang (line 1: `#!/usr/bin/env node`) is similarly stripped, but since `index.js` is not the npm bin target, it is lower priority.
- **Assumption:** The npm bin wrapper mechanism (`"px": "px.js"`) works correctly when `px.js` has a shebang — npm will create an executable wrapper that invokes `node px.js`.
